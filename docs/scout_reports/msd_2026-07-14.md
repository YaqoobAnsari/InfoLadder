## Split contents — IMPORTANT

The zips ship linked modalities per plan, keyed by integer id:

| folder | file | content |
|---|---|---|
| `graph_out/` | `<id>.pickle` | `networkx.Graph`; **the rich modality** (train only) |
| `graph_in/` | `<id>.pickle` | `networkx.Graph`; zoning input only (`zoning_type`) |
| `struct_in/` | `<id>.npy` | `(512,512,3) float16` structure raster (input) |
| `full_out/` | `<id>.npy` | full plan raster (train only) |

- **The TEST split is input-only**: it has `graph_in` + `struct_in` but **no
  `graph_out`** — MSD is a floor-plan *generation* benchmark, so the geometry
  ground truth is withheld for test. `graph_in` nodes carry only `zoning_type`
  (no geometry/centroid/room_type). **⇒ only the ~4,167 TRAIN plans are usable as
  rich SpectrumGraphs.** We stream from the train zip.

## graph_out format (dissected from real pickles)

`networkx.Graph`, integer node keys, `G.graph == {}` (no id → native id = filename).

- **Node attrs**: `geometry` = `list[(x, y)]` polygon exterior ring in **meters**;
  `room_type` = **int index into** `ROOM_NAMES` = [Bedroom, Livingroom, Kitchen,
  Dining, Corridor, Stairs, Storeroom, Bathroom, Balcony, Structure, Door,
  Entrance Door, Window] (only indices 0–8, the *areas*, ever appear as nodes);
  `centroid` = `torch.Tensor([x, y])`.
- **Edge attr**: `connectivity` ∈ {`door`, `passage`, `entrance`}. `passage` = the
  two areas touch with no wall between (open access, dist < 0.04 m); `door`/
  `entrance` = a door/front-door polygon lies within 0.05 m of both areas. The
  access graph is **undirected**; walls/windows are NOT nodes or edges.
- **Zone labels**: `graph_in` stores `zoning_type` (int index into `ZONING_NAMES`
  = [Zone1..Zone4, Structure, Door, Entrance Door, Window]). The zone is a
  **deterministic function of room type** (constants.py `ZONING_ROOMS`): Bedroom→
  Zone1; Living/Kitchen/Dining/Corridor→Zone2; Stairs/Storeroom/Bathroom→Zone3;
  Balcony→Zone4. So we derive zone from `graph_out.room_type` — no need to read
  `graph_in` at all.

## Mapping to SpectrumGraph (the loader)

Build the richest level MSD supports (R4), then `forget()` derives R3/R2/R1/R0
(strict refinement holds by construction, claim C1):

- **spaces** → `room` (or `corridor` for the Corridor type) nodes; `area` = shoelace
  of the polygon (m²), `centroid` (m), `label` = room type. Room nodes carry **no
  `attrs`** (else forget→R3 would fail validation — zone info must not live on rooms).
- **edges** → `tau`: door|entrance → `door`, passage → `corridor-link`; **`delta`
  = `both`** for every edge (MSD ships no direction).
- **near-R4 zone forest** (`zone_mode`):
  - `category` (default, = "shipped zone label group"): one `zone` node per
    Zone1–4 present; room→zone containment; zone `attrs` = group, member count,
    room-type histogram, total area.
  - `apartment`: one `zone` node per spatial unit = connected component of the
    access graph after **cutting entrance-door edges** (a front door separates an
    apartment from the common stair/corridor). Recovers real apartments.

## Two limitations to weigh (flagged for DECISIONS)

1. **Category zone leaks.** Because zone = f(room_type), a category-zone target is
   readable from the room label at *every* level, not just R4 — it cannot serve as
   the R4-exclusive Y_zone probe (plan §8 leakage). **`zone_mode='apartment'`
   fixes this**: apartment membership needs the access-graph structure (entrance
   cuts), so it is genuinely recoverable only at R4. Recommend adopting apartment
   grouping for the S9/zone experiment; category grouping is fine as a coarse
   room-function target. MSD ships **no per-apartment id** in the graph, so the
   access-cut is the only route to spatial units.
2. **No access direction.** MSD's access graph is undirected → all `delta='both'`,
   i.e. the R2→R3 increment is EMPTY for MSD. MSD supports the R0/R1/R2 and (near-)
   R4 probes richly but contributes nothing to the R3 (direction) probe — direction
   must come from the IFC/gold lane (Gate-d / §7).
